# Part 3 — SRv6

## 3.0 The Problem: How Do Overlays Reach Each Other?

At this point, IS-IS has given us a working underlay: every PE can reach every
other PE's loopback and locator prefix. But the overlay services (VPN routes,
VXLAN tunnels) need a **transport mechanism** — a way to wrap a customer packet
inside a transit-friendly envelope and deliver it to the right PE.

In traditional networks, this transport was **MPLS** (Multi-Protocol Label
Switching). A PE would push an MPLS label onto a packet, transit routers would
swap labels hop-by-hop, and the destination PE would pop the label and deliver
the packet to the right VRF. MPLS worked well but had a significant drawback:
every transit router needed to understand MPLS. You couldn't send MPLS packets
over a network that only understood IP.

**SRv6 (Segment Routing over IPv6)** solves this elegantly: instead of MPLS
labels, it uses **IPv6 addresses** as forwarding instructions. Since every
router already knows how to forward IPv6 packets (that's what the underlay
does!), you get transport for free. Transit routers don't need to understand
SRv6 — they just perform normal IPv6 routing.

## 3.1 What is Segment Routing? (SR Primer)

Before diving into SRv6, you need to understand **Segment Routing (SR)** as a
paradigm. SR is an architecture — a way of thinking about forwarding packets
through a network — that has two concrete implementations (SR-MPLS and SRv6).
The core ideas apply to both.

### The Problem SR Was Invented to Solve

Traditional service provider networks used **MPLS** (Multi-Protocol Label
Switching) for traffic transport. MPLS itself is simple: routers push short
numeric labels onto packets, and each transit router swaps one label for another
to move the packet toward its destination. But MPLS has a control-plane problem
— it needs **additional protocols** to distribute labels and set up paths:

- **LDP (Label Distribution Protocol):** Every router in the network runs LDP to
  discover neighbors and exchange label bindings. Each router-to-router link
  needs an LDP session. For a network of 100 routers, that's hundreds of LDP
  sessions to maintain, each with its own state, timers, and failure modes.
- **RSVP-TE (Resource Reservation Protocol — Traffic Engineering):** If you need
  traffic-engineered paths (explicit routes that don't follow the shortest IGP
  path), you need RSVP-TE. Each path requires an end-to-end signaled tunnel
  with per-hop state maintained at every router along the path. Thousands of
  tunnels means thousands of state entries at every transit router.

The result: three protocols (IGP + LDP + RSVP-TE) running simultaneously, each
with its own adjacencies, state tables, and failure modes. When something
breaks, you're debugging interactions between three complex distributed systems.

**Segment Routing eliminates LDP and RSVP-TE entirely.** It achieves the same
outcomes — label-based forwarding and traffic engineering — using only the IGP
(IS-IS or OSPF) for control-plane distribution, plus a fundamentally different
forwarding model.

### The Core Idea: Source Routing

Traditional hop-by-hop routing works like passing a letter through a mail
chain: you hand it to the first person, they decide who to give it to next,
that person decides the next handoff, and so on. Each hop makes an independent
decision. If you want the letter to take a specific path, you'd have to
instruct every person in the chain individually.

Segment Routing flips this: the **source** (the first router that touches the
packet) writes the complete set of forwarding instructions directly into the
packet header. Each instruction is called a **segment**. Transit routers
don't need any special protocol state — they just read the next instruction,
execute it, and forward.

Think of it like putting an ordered list of street addresses on a package:
"Go to 100 Main St, then to 42 Oak Ave, then to 7 Elm Dr." Each handler in
the chain reads the next address on the list and sends the package there.
No one needs to know the full delivery plan — they just handle the next hop.

### Segments: The Fundamental Building Block

A **segment** is an instruction. It tells a router what to do with a packet.
Segments come in several types:

**Prefix Segment (Node SID):**
The most common segment type. It says "forward this packet toward node X using
the shortest IGP path." Every router in the network is assigned a unique node
SID, and that SID is advertised through the IGP (IS-IS or OSPF). When a router
encounters a prefix segment, it just forwards toward that node — exactly like
normal routing, but the instruction is explicitly encoded in the packet.

```
Node SID 16 → "Forward toward PE1 via the shortest path"
Node SID 17 → "Forward toward PE2 via the shortest path"
Node SID 18 → "Forward toward PE3 via the shortest path"
```

**Adjacency Segment (Adj-SID):**
A locally significant instruction that says "send this packet out a specific
link." Adjacency SIDs are assigned per-link and are only meaningful to the
router that owns them. They enable **traffic engineering** — you can force
a packet to take a specific link even if it's not the shortest path.

```
Adj-SID 24001 on PE1 → "Send out the link to Spine (even if a shorter path exists)"
```

**Service Segment:**
An instruction that says "perform a service function on this packet" —
for example, "decapsulate and deliver to VRF CMN." Service segments are how
SR integrates with overlay services like L3VPN and EVPN. In SRv6, these are
called **End.DT4**, **End.DT6**, etc.

### Segment Lists (Stacking Instructions)

The power of SR is that segments can be **stacked**. The source encodes an
ordered list of segments in the packet header, and each router along the path
processes the next segment in the list.

For simple forwarding (shortest path from A to B), you only need one segment:

```
Segment List: [Node-SID-of-B]
→ "Go to B" — every transit router forwards toward B's node SID
```

For traffic engineering (forcing a specific path through the network), you
stack multiple segments:

```
Segment List: [Node-SID-of-C, Adj-SID-of-link-C→D, Node-SID-of-B]
→ "First go to C, then take C's link to D, then go to B"
```

The source router pushes this list onto the packet. Each router processes the
top segment, and once that segment's instruction is fulfilled (e.g., the packet
has arrived at node C), the next segment becomes active. This continues until
the list is empty and the packet reaches its final destination.

```
Step 1 (at Source):  Segment List: [C, C→D, B]    Active: C
                     → Forward toward C on shortest path

Step 2 (at C):       Segment List: [C, C→D, B]    Active: C→D
                     → Send out the specific link C→D

Step 3 (at D):       Segment List: [C, C→D, B]    Active: B
                     → Forward toward B on shortest path

Step 4 (at B):       Segment List empty → deliver packet
```

No router other than the source needed to know the full path. C just executed
its instruction. D just forwarded toward B normally. This is the essence of
source routing: the ingress decides, transit just executes.

### SR-MPLS: Segment Routing over the MPLS Data Plane

The first implementation of SR uses MPLS labels as segment identifiers. Each
segment is a 20-bit MPLS label, and segment lists are encoded as a stack of
MPLS labels pushed onto the packet.

```
┌──────────────────────────────────┐
│ MPLS Label Stack                 │
│   Top:    Label 16 (Node SID PE2)│  ← processed first
│   Bottom: Label 3  (VPN label)   │  ← processed at PE2
├──────────────────────────────────┤
│ Inner IP Packet                  │
│   src=10.0.0.1  dst=10.0.0.2    │
└──────────────────────────────────┘
```

Each transit router pops/swaps the top label and forwards. The label stack
shrinks as the packet moves through the network. When the last label is
reached, the destination router performs the service action (e.g., VPN
decapsulation).

**SR-MPLS advantages:**
- Minimal change from traditional MPLS networks — same data plane, same label
  forwarding hardware
- Eliminates LDP (node SIDs are distributed via IGP)
- Eliminates RSVP-TE (traffic engineering via adjacency SID stacking)

**SR-MPLS limitations:**
- Still requires MPLS-capable forwarding on every transit router
- MPLS labels are 20 bits — limited instruction space
- Labels are locally significant (same label number means different things on
  different routers), making debugging harder

### SRv6: Segment Routing over the IPv6 Data Plane

The second (and newer) implementation of SR uses **IPv6 addresses** as segment
identifiers. Instead of MPLS labels, each segment is a 128-bit IPv6 address.
Segment lists are encoded in an **IPv6 extension header** called the Segment
Routing Header (SRH).

The key insight: since every segment is a valid IPv6 address, transit routers
don't need to understand SR at all — they just perform standard IPv6 routing.
The network only needs to support IPv6, which every modern router and OS
already does.

```
┌──────────────────────────────────────────┐
│ IPv6 Header                              │
│   src = fd00:30:16::  (PE1)              │
│   dst = fd00:30:17:3:: (active segment)  │
├──────────────────────────────────────────┤
│ SRH (Segment Routing Header)             │
│   Segments Left: 0                       │
│   Segment List:                          │
│     [0] fd00:30:17:3:: (End.DT4 on PE2) │
├──────────────────────────────────────────┤
│ Inner IPv4 Packet                        │
│   src=10.55.96.149  dst=10.55.96.153     │
└──────────────────────────────────────────┘
```

When a packet has only one segment (the common case in this lab), the SRH can
be omitted entirely — the destination IPv6 address IS the segment. This is
called **reduced encapsulation** and it's what micro-SID (uSID) mode enables.

### The Segment Routing Header (SRH)

The SRH is an IPv6 Routing Extension Header (type 4) that carries the segment
list. Its key fields:

| Field            | Purpose                                                         |
| ---------------- | --------------------------------------------------------------- |
| Segments Left    | Index into the segment list — how many segments remain          |
| Segment List     | Ordered array of 128-bit IPv6 addresses (the full path)         |
| Last Entry       | Index of the last segment in the list                           |

**How SRH processing works:**

1. The source builds the SRH with the full segment list, ordered bottom-to-top
   (first segment at the highest index). It sets `Segments Left` to point at
   the first segment and copies that segment into the IPv6 Destination Address.
2. Transit routers forward based on the Destination Address — standard IPv6
   routing. They don't look at the SRH at all.
3. When the packet reaches the node identified by the active segment, that node
   decrements `Segments Left`, copies the next segment from the list into the
   Destination Address, and forwards again.
4. When `Segments Left` reaches 0, the current destination is the final
   segment. That node performs the endpoint behavior (e.g., decapsulate into a
   VRF).

```
Packet at source (PE1):
  IPv6 dst = fd00:30:1:17:3::    (first segment: via spine to PE2)
  SRH Segments Left = 1
  SRH Segment List: [0] fd00:30:17:3::   [1] fd00:30:1:17:3::

At spine (processes segment [1]):
  IPv6 dst = fd00:30:1:17:3::    matches spine → shift uSID → new dst = fd00:30:17:3::
  SRH Segments Left = 0

At PE2 (processes segment [0]):
  IPv6 dst = fd00:30:17:3::      matches local SID → End.DT4 → decap to VRF CMN
```

In this lab's star topology, packets rarely need multi-segment paths (one hop
through the spine gets you anywhere), so the SRH is usually absent and the
single segment lives directly in the IPv6 Destination Address.

### SR-MPLS vs SRv6: Comparison

| Aspect                    | SR-MPLS                        | SRv6                                         |
| ------------------------- | ------------------------------ | -------------------------------------------- |
| Segment ID size           | 20 bits (MPLS label)           | 128 bits (IPv6 address)                      |
| Transit router requires   | MPLS forwarding                | Standard IPv6 routing                        |
| Segment list encoding     | MPLS label stack               | SRH (IPv6 extension header)                  |
| Instruction space         | Limited (20-bit label)         | Rich (128 bits encode node + function + arg) |
| Network programmability   | Label = destination only       | SID = destination + action + arguments       |
| Hardware support          | Mature (decades of MPLS ASICs) | Newer (IPv6 extension header processing)     |
| Linux kernel native       | Requires MPLS kernel modules   | Native since kernel 4.10                     |
| Overhead per segment      | 4 bytes                        | 16 bytes (or 0 with uSID compression)        |

The critical difference for this lab: SRv6 SIDs encode **both the destination
AND the action** in a single address. `fd00:30:17:3::` simultaneously means
"go to PE2" (the `/48` prefix routes you there) AND "decapsulate into VRF CMN"
(the function `3` triggers `End.DT4`). In SR-MPLS, you'd need two separate
labels — one for the transport path and one for the VPN service.

### Why This Architecture Uses SRv6

1. **IPv6-native** — the underlay is already IPv6, so SRv6 is a natural fit
2. **No MPLS dependency** — transit routers (the spine in this lab) just forward IPv6
3. **Programmability** — SRv6 SIDs encode both destination AND action (e.g.,
  "decap into VRF CMN on PE2"), reducing the number of protocol interactions
4. **Linux kernel support** — SRv6 is implemented natively in the Linux kernel
  (since 4.10), no special software needed on the data plane
5. **Simplified control plane** — no LDP, no RSVP-TE, no MPLS label allocation.
  IS-IS distributes locators, BGP distributes SRv6 SIDs for services — two
  protocols, each with a clean, well-defined role

## 3.2 SRv6 Concepts

### Locator

A `/48` IPv6 prefix assigned to each node. IS-IS advertises it so the entire
network knows how to reach each node. Think of it as the node's "address
block" — all SRv6 instructions for that node live within its locator.

```
fd00:30:16::/48   ← PE1's locator (any address starting with fd00:0030:0016 belongs to PE1)
fd00:30:17::/48   ← PE2's locator
fd00:30:18::/48   ← PE3's locator
fd00:30:1::/48    ← spine's locator
```

Transit routers don't need to know what's inside the locator — they just match
the /48 prefix and forward toward the owning node. This is standard IPv6
longest-prefix matching.

### SID (Segment Identifier)

A specific IPv6 address within a locator that encodes a forwarding instruction.
Different SIDs within the same locator trigger different **behaviors**
(actions) at the destination node:

```
fd00:30:16:3::   ← "decapsulate and deliver to VRF CMN on PE1" (End.DT4)
fd00:30:16:1::   ← "forward to next segment" (End behavior)
```

The key insight: the SID is a regular IPv6 address. Transit routers route it
normally using the /48 prefix. But when the packet arrives at the **owning
node** (PE1, in this case), that node recognizes the full address as one of its
SIDs and performs the associated action. The destination is also the
instruction.

### SRv6 Locator Structure (How the Address Encodes Instructions)

```
fd00:30:16::/48
├── block-len 32    → fd00:0030 (the first 32 bits identify the SRv6 domain)
├── node-len 16     → 0016 (the next 16 bits identify the specific node)
└── function (remaining 80 bits encode the action)

fd00:30:16:0003:0000:0000:0000:0000
│          │    │
│          │    └── argument (unused here — can carry additional context)
│          └── function = 3 (VRF CMN decap — assigned by FRR automatically)
└── locator = PE1 (fd00:30:16)
```

Breaking it down:

- **Block** (32 bits): `fd00:0030` — identifies this SRv6 domain. All nodes in
the fabric share this block prefix.
- **Node** (16 bits): `0016` — uniquely identifies PE1 within the domain. PE2
is `0017`, PE3 is `0018`, the spine is `0001`.
- **Function** (16 bits): `0003` — the operation to perform. FRR auto-assigns
function IDs when BGP allocates SIDs for VRFs. Function 3 might mean "End.DT4
for VRF CMN" — the exact number is dynamically assigned but consistent.
- **Arguments** (remaining 64 bits): Zero in this lab. Can carry additional
context in more complex deployments.

### Micro-SID (uSID)

This lab uses `behavior usid` (micro-SID), which is an optimization. In
traditional SRv6, if a packet needs to traverse multiple segments, each
segment is a full 128-bit IPv6 address listed in a Segment Routing Header
(SRH). With micro-SID, multiple short instructions are packed into a single
128-bit IPv6 destination address. Each micro-SID is 16 bits (the node-len
portion), and the router shifts through them as the packet traverses the
network.

For this lab's simple star topology, micro-SID isn't strictly necessary (packets
usually have only one segment). But it's the standard SRv6 deployment model
and avoids SRH overhead entirely for single-segment paths, which means less
encapsulation overhead and better performance.

## 3.3 SRv6 Behaviors (Actions)

When a packet arrives at its SRv6 destination, the node looks up the SID in
its local SID table and performs the associated **behavior**. The most
important behaviors in this architecture:


| Behavior   | Full Name                                         | What It Does                                                                                                                                        |
| ---------- | ------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `End`      | Endpoint                                          | Basic transit: advance to the next segment in the SID list. Used when a packet has multiple segments.                                               |
| `End.DT4`  | Endpoint with Decapsulation and IPv4 Table lookup | Strip the outer IPv6 header, take the inner IPv4 packet, and route it using a specific VRF's routing table. This is how VPN traffic gets delivered. |
| `End.DT6`  | Endpoint with Decapsulation and IPv6 Table lookup | Same as End.DT4 but for inner IPv6 packets.                                                                                                         |
| `End.DT46` | Endpoint with Decap and Dual-Stack Table lookup   | Handles both IPv4 and IPv6 inner packets.                                                                                                           |


In this lab, the critical behavior is **End.DT4**: when PE2 receives a packet
destined for `fd00:30:17:3::`, it:

1. Recognizes `fd00:30:17:3::` as its own SID
2. Looks up the behavior: End.DT4, VRF table 4 (CMN)
3. Strips the outer IPv6 header, revealing the inner IPv4 packet
4. Routes the inner packet using CMN VRF's routing table

This is how SRv6 delivers traffic to the correct VRF on the correct PE.

## 3.4 How SRv6 Encapsulation Works

When PE1 needs to send a VRF packet to PE2, the process is:

```
Step 1 — Original packet (inside CMN VRF):
┌─────────────────────────────────────────┐
│ IP: src=10.55.96.149  dst=10.55.96.153  │
│ (CMN VRF traffic — e.g., VTEP trying    │
│  to reach a remote VTEP)                │
└─────────────────────────────────────────┘

Step 2 — CMN VRF routing table lookup:
  Route: 10.55.96.152/30 via SRv6 SID fd00:30:17:3::
  Action: encapsulate with SRv6

Step 3 — After SRv6 encapsulation at PE1:
┌─────────────────────────────────────────┐
│ Outer IPv6:                             │
│   src = fd00:30:16::  (PE1's locator)   │
│   dst = fd00:30:17:3:: (PE2's VRF SID)  │
├─────────────────────────────────────────┤
│ Inner IPv4:                             │
│   src=10.55.96.149  dst=10.55.96.153    │
│   (the original packet, unchanged)      │
└─────────────────────────────────────────┘
```

The outer IPv6 header is a standard IPv6 packet. Any IPv6 router can forward
it. The inner packet is completely opaque to transit routers.

**Transit at the spine:**

- The spine receives the packet and looks at the destination: `fd00:30:17:3::`
- Its routing table has: `fd00:30:17::/48` via eth2 (learned from IS-IS)
- The spine forwards the packet out eth2 toward PE2
- The spine never looks at the inner packet — it's just forwarding IPv6

**Arrival at PE2:**

- PE2 receives the packet destined for `fd00:30:17:3::`
- The kernel matches this against the local SID table
- Action: End.DT4, VRF table 4 (CMN)
- The outer IPv6 header is stripped
- The inner IPv4 packet (10.55.96.149 → 10.55.96.153) is delivered to the
CMN VRF routing table
- CMN knows 10.55.96.152/30 is connected on `ve-pe-cmn-l2` → packet delivered

## 3.5 SRv6's Role in This Solution

SRv6 is the **transport glue** between the underlay and the overlays:

- **For BGP L3VPN:** When PE1 learns (via BGP) that PE2's CMN VRF has subnet
10.55.96.152/30, the BGP route includes PE2's SRv6 SID (`fd00:30:17:3::`).
When PE1 needs to send traffic to that subnet, it SRv6-encapsulates the
packet using the SID. The underlay (IS-IS) routes the SRv6 packet to PE2.
PE2 decapsulates and delivers to CMN VRF. SRv6 is the "how" — BGP provides
the "what" (routing information) and SRv6 provides the "how" (encapsulation
and delivery).
- **For VXLAN/EVPN:** VTEP-to-VTEP traffic (VXLAN-encapsulated frames) travels
through the CMN VRF. The CMN VRF uses SRv6 to reach remote subnets. So VXLAN
traffic is actually double-encapsulated: the original Ethernet frame is
wrapped in VXLAN, then the VXLAN packet is wrapped in SRv6. This double
encapsulation is why jumbo frames are critical.
- **Compared to MPLS:** In an MPLS network, you'd need LDP or RSVP-TE to
distribute labels, and every transit router would need MPLS forwarding
enabled. With SRv6, the transit routers (the spine) just do normal IPv6 routing.
The "label" (SID) is the destination address itself. Simpler, fewer
protocols, fewer things to break.

## 3.6 SRv6 Kernel Requirements

SRv6 is implemented in the Linux kernel (not in FRR). FRR just programs the
kernel with SRv6 routes and SID behaviors. These sysctls enable the kernel's
SRv6 processing:

```bash
sysctl -w net.ipv6.conf.all.seg6_enabled=1      # global: enable SRv6 processing
sysctl -w net.ipv6.conf.eth1.seg6_enabled=1      # per-interface: enable on eth1
sysctl -w net.ipv6.seg6_flowlabel=1               # copy flow info for ECMP hashing
```

**Why per-interface enabling?** SRv6 processing has a small CPU cost for every
IPv6 packet (the kernel checks if the destination matches a local SID). You
only want this on interfaces that actually carry SRv6 traffic (the fabric
interface), not on every interface on the system.sudo ip netns exec pe-router ip route show   

**What the kernel installs for SRv6:**

When FRR allocates an SRv6 SID (e.g., for VRF CMN), it programs a route like:

```
fd00:30:16:3:: encap seg6local action End.DT4 vrftable 4 dev CMN
```

This is a kernel route (not an FRR route). It tells the kernel: "when a packet
arrives for `fd00:30:16:3::`, perform `End.DT4` — decapsulate and look up the
inner packet in VRF table 4." The `seg6local` encapsulation type is the
kernel's way of representing SRv6 local SID behaviors.

## 3.7 Lab Walkthrough: Verify SRv6

```bash
# SSH into PE1
ssh clab@clab-srv6-lab-pe1

# Check SRv6 locator is active:
sudo podman exec frr-pe-router vtysh -c 'show segment-routing srv6 locator'
# Name   ID  Prefix             Status
# MAIN    1  fd00:30:16::/48    Up

# Check which SIDs were allocated:
sudo podman exec frr-pe-router vtysh -c 'show segment-routing srv6 sid'
# This shows every SID FRR has allocated and what behavior it maps to

# See SRv6 routes in the kernel:
sudo ip netns exec pe-router ip -6 route show | grep seg6
```

The kernel routes will show entries like:

```
fd00:30:16:3:: encap seg6local action End.DT4 vrftable 4 dev CMN
```

This means: "when a packet arrives with destination `fd00:30:16:3::`, strip
the outer header and route the inner packet using VRF table 4 (CMN)." This is
the `End.DT4` behavior — **End** (endpoint, meaning this is the final
destination for the SRv6 path), **DT** (Decapsulation + Table lookup — remove
the SRv6 header and look up the inner packet), **4** (IPv4 — the inner packet
is IPv4). Note: the "4" in End.DT4 refers to the IP version of the inner
packet, NOT the VRF table number — it's a coincidence that the CMN VRF table
is also 4.
